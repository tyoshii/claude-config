# /teamdev <タスク説明>

worktree を作成し、エージェントチームを編成して自律的に開発を進める。

ユーザーはタスクを投げたら待つだけ。
エージェントたちが調査・設計・実装・テストを分担して進める。

## 実行手順

### 1. ブランチ名の生成

タスク説明からブランチ名を自動生成する：

- 形式: `team/<短い英語の要約>`
- 例:
  - 「認証機能を追加して」→ `team/add-auth`
  - 「パフォーマンス改善」→ `team/improve-performance`

### 2. worktree の作成

```bash
eval "$(command vibe start <branch>)" && pwd
```

- 出力の最後の行が新しい作業ディレクトリのパス
- 以降すべてのファイル操作はこのパスを基準にする

### 3. タスクの分解

タスク説明を分析し、並列に進められるサブタスクに分解する。

分解の方針：

- **依存関係がないもの**は並列に進める
- **依存関係があるもの**はフェーズを分けて順次実行
- 各サブタスクは 1 エージェントが完結できる粒度にする

### 4. フェーズ 1 — 調査

まず Agent ツール（`subagent_type: Explore`）を**並列に**使い、以下を把握する：

- 既存コードの構造と関連ファイル
- 使われている技術スタック・パターン
- テストの有無と実行方法
- CLAUDE.md やプロジェクト固有のルール

調査結果をもとに、実装計画を組み立てる。

### 5. フェーズ 2 — 実装

Agent ツール（`subagent_type: general-purpose`）を使い、サブタスクごとにエージェントを起動する。
**同時にレビュー役エージェントも起動する。**

#### エージェントへの指示テンプレート

各実装エージェントには以下を伝える：

```
## 作業ディレクトリ
<worktree の絶対パス>

## 担当タスク
<サブタスクの説明>

## コンテキスト
- プロジェクト概要: <調査で得た情報>
- 関連ファイル: <ファイルパスの一覧>
- 既存パターン: <コーディング規約・スタイル>

## ルール
- ファイルの読み書きは必ず作業ディレクトリの絶対パスを使う
- git 操作（add, commit）は行わない
- 他のエージェントの担当範囲のファイルは変更しない
```

#### レビュー役エージェント

実装エージェントと**並列で**レビュー役エージェント（`subagent_type: general-purpose`）を 1 つ起動する。

レビュー役への指示：

```
## 役割
あなたはコードレビュアーです。実装エージェントたちが作業中のコードを継続的にレビューしてください。

## 作業ディレクトリ
<worktree の絶対パス>

## タスクの全体像
<元のタスク説明と実装計画の要約>

## レビュー手順
1. 30秒ほど待ってから、変更されたファイルを確認する
2. 以下の観点でレビューする：
   - バグや論理エラー
   - セキュリティ上の問題
   - 既存コードとのスタイル・パターンの不一致
   - エッジケースの考慮漏れ
   - ファイル間の整合性
3. 問題を発見した場合、指摘事項をまとめる
4. 1-2 の間隔をおいてもう一度チェックし、新しい変更があればレビューする
5. 最終的なレビュー結果をまとめて返す

## ルール
- ファイルは読み取りのみ。変更は行わない
- git 操作は行わない
- レビューは 3-5 回繰り返す（実装の進行に合わせて）
- 最終レポートには「問題なし」の項目は含めず、指摘事項のみ記載する

## 出力形式
最終結果を以下の形式で返す：
### 中間レビュー指摘（実装中に発見）
- [ファイルパス:行] 指摘内容

### 最終レビュー
- [ファイルパス:行] 指摘内容

指摘がない場合は「レビュー完了：指摘事項なし」とだけ返す。
```

#### 並列実行の制約

- 同じファイルを複数の実装エージェントが編集しないようにする
- ファイルの担当が重なる場合はフェーズを分ける
- 各実装エージェントには担当ファイルを明示する
- レビュー役は読み取り専用で、ファイル変更を行わない

### 6. フェーズ 3 — 統合・検証

全エージェントの作業完了後：

1. **結果の確認** — 変更されたファイルを一覧し、整合性を確認
2. **レビュー指摘の反映** — レビュー役エージェントの指摘があれば対応する
3. **テスト実行** — テストがあれば実行する
4. **修正** — テスト失敗やコンフリクトがあれば修正する

### 7. PR 作成

コミットして PR を作成する：

1. `git add` で変更ファイルをステージング（機密ファイルを除く）
2. `/commit` の手順に従ってコミットメッセージを作成・コミット
3. `git push -u origin <ブランチ名>` でプッシュ
4. `gh pr create` で PR を作成

PR 本文の形式：
```markdown
## Summary
- 変更点を箇条書きで簡潔に

## Test plan
- テスト方法や確認事項を箇条書きで
```

### 8. PR レビュー

PR 作成後、Agent ツール（`subagent_type: general-purpose`）で**レビューエージェント**を起動する。

レビューエージェントへの指示：

```
## 役割
あなたは PR のコードレビュアーです。

## PR 情報
- PR URL: <作成した PR の URL>
- リポジトリ: <リポジトリのパス>
- ブランチ: <ブランチ名>

## 手順
1. `gh pr diff <PR番号>` で差分を取得
2. 変更内容を詳細にレビューする：
   - バグや論理エラー
   - セキュリティ上の問題
   - コードスタイル・パターンの一貫性
   - エッジケースの考慮漏れ
   - テストの十分性
3. 指摘事項を PR にコメントとして投稿する

## コメント投稿方法

**全体コメント（総評）：**
gh pr comment <PR番号> --body "<レビュー総評>"

**ファイル単位のレビューコメント：**
gh api repos/<owner>/<repo>/pulls/<PR番号>/reviews \
  --method POST \
  -f body="コードレビュー" \
  -f event="COMMENT" \
  -f 'comments[][path]=<ファイルパス>' \
  -f 'comments[][position]=<diff内の行位置>' \
  -f 'comments[][body]=<コメント内容>'

※ `position` は diff のハンク内での行番号（1始まり）。
正確な position の算出が難しい場合は、全体コメントにファイル名と行番号を含めて投稿する。

## ルール
- 軽微なスタイルの指摘は省略してよい
- 重要な問題には 🔴、改善提案には 🟡、良い点には 🟢 のプレフィックスをつける
- git push や PR のマージは行わない
```

### 9. 完了報告

```
## チーム作業完了

### タスク
<元のタスク説明>

### 実施内容
- <サブタスク1>: 完了
- <サブタスク2>: 完了
- ...

### 変更ファイル
<git diff --stat の結果>

### テスト結果
<テスト実行結果の要約>

### レビュー結果
<レビュー指摘の要約、または「指摘事項なし」>

### PR
<PR の URL>

### 次のステップ
- PR のレビューコメントを確認
- 必要に応じて修正対応
- `vibe clean` で worktree を削除
```

## エージェント編成の例

### 小規模タスク（1-2 ファイル）

エージェントは 1 つで十分。分割しない。

### 中規模タスク（3-5 ファイル）

```
[Explore] 調査 ──→ [Agent A] フロントエンド ──→ /commit & /pr ──→ [Reviewer] PR レビュー
                 ├→ [Agent B] バックエンド
                 ├→ [Reviewer] 中間レビュー
                 └→ 統合・テスト
```

### 大規模タスク（6+ ファイル）

```
[Explore x2] 調査 ──→ [Agent A] 機能実装    ──→ /commit & /pr ──→ [Reviewer] PR レビュー
                    ├→ [Agent B] API/データ層
                    ├→ [Agent C] テスト作成
                    ├→ [Reviewer] 中間レビュー
                    └→ 統合・検証
```

## 注意事項

- 実装エージェントの数は多くても 3-4 まで。増やしすぎるとコンフリクトが増える
- レビュー役は常に 1 エージェント（実装中の中間レビュー + PR レビュー）
- 各エージェントに worktree の**絶対パス**を必ず渡す
- エージェントが失敗した場合は、そのサブタスクだけリトライする
- worktree 内での作業なので、メインの作業ディレクトリには影響しない
- `gh` CLI がインストール・認証済みであることが前提
