# /teamdev <タスク説明>

worktree を作成し、エージェントチームを編成して自律的に開発を進める。

ユーザーはタスクを投げたら待つだけ。
エージェントたちが調査・設計・実装・テストを分担して進める。

## 実行手順

### 1. ブランチ名の生成

タスク説明からブランチ名を自動生成する：

- 形式: `team/<短い英語の要約>`
- 例:
  - 「認証機能を追加して」→ `team/add-auth`
  - 「パフォーマンス改善」→ `team/improve-performance`

### 2. worktree の作成

```bash
eval "$(command vibe start <branch>)" && pwd
```

- 出力の最後の行が新しい作業ディレクトリのパス
- 以降すべてのファイル操作はこのパスを基準にする

### 3. タスクの分解

タスク説明を分析し、並列に進められるサブタスクに分解する。

分解の方針：

- **依存関係がないもの**は並列に進める
- **依存関係があるもの**はフェーズを分けて順次実行
- 各サブタスクは 1 エージェントが完結できる粒度にする

### 4. フェーズ 1 — 調査

まず Agent ツール（`subagent_type: Explore`）を**並列に**使い、以下を把握する：

- 既存コードの構造と関連ファイル
- 使われている技術スタック・パターン
- テストの有無と実行方法
- CLAUDE.md やプロジェクト固有のルール

調査結果をもとに、実装計画を組み立てる。

### 5. フェーズ 2 — 実装

Agent ツール（`subagent_type: general-purpose`）を使い、サブタスクごとにエージェントを起動する。

#### エージェントへの指示テンプレート

各エージェントには以下を伝える：

```
## 作業ディレクトリ
<worktree の絶対パス>

## 担当タスク
<サブタスクの説明>

## コンテキスト
- プロジェクト概要: <調査で得た情報>
- 関連ファイル: <ファイルパスの一覧>
- 既存パターン: <コーディング規約・スタイル>

## ルール
- ファイルの読み書きは必ず作業ディレクトリの絶対パスを使う
- git 操作（add, commit）は行わない
- 他のエージェントの担当範囲のファイルは変更しない
```

#### 並列実行の制約

- 同じファイルを複数エージェントが編集しないようにする
- ファイルの担当が重なる場合はフェーズを分ける
- 各エージェントには担当ファイルを明示する

### 6. フェーズ 3 — 統合・検証

全エージェントの作業完了後：

1. **結果の確認** — 変更されたファイルを一覧し、整合性を確認
2. **テスト実行** — テストがあれば実行する
3. **修正** — テスト失敗やコンフリクトがあれば修正する

### 7. 完了報告

```
## チーム作業完了

### タスク
<元のタスク説明>

### 実施内容
- <サブタスク1>: 完了
- <サブタスク2>: 完了
- ...

### 変更ファイル
<git diff --stat の結果>

### テスト結果
<テスト実行結果の要約>

### 次のステップ
- `/commit push` でコミット＆プッシュ
- `gh pr create` で PR 作成
- `vibe clean` で worktree を削除
```

## エージェント編成の例

### 小規模タスク（1-2 ファイル）

エージェントは 1 つで十分。分割しない。

### 中規模タスク（3-5 ファイル）

```
[Explore] 調査 ──→ [Agent A] フロントエンド
                 ├→ [Agent B] バックエンド
                 └→ 統合・テスト
```

### 大規模タスク（6+ ファイル）

```
[Explore x2] 調査 ──→ [Agent A] 機能実装
                    ├→ [Agent B] API/データ層
                    ├→ [Agent C] テスト作成
                    └→ 統合・検証
```

## 注意事項

- エージェントの数は多くても 3-4 まで。増やしすぎるとコンフリクトが増える
- 各エージェントに worktree の**絶対パス**を必ず渡す
- git 操作はすべて完了後にユーザー判断で行う
- エージェントが失敗した場合は、そのサブタスクだけリトライする
- worktree 内での作業なので、メインの作業ディレクトリには影響しない
