# /overnight

夜間自走モードに入る。ユーザー不在の間、主体的に作業を進める。

通常の制約よりも自律性を優先する。

## 基本方針

- ユーザーは即時に応答しない前提で進める
- 判断が必要な場合は、合理的な仮定を置いて進めてよい
- 途中で止まらず、一定の成果が残るところまで進める

## ブランチ運用

作業は必ず専用ブランチで行う。

```
overnight/yyyy-mm-dd-topic
```

例: `overnight/2025-01-17-auth-refactor`

**禁止**: main / develop など既存ブランチへの直接コミット

## コミット方針

- 作業途中でも、意味のある単位でコミットしてよい
- コミットメッセージは日本語で記述
- 調査・整理・リファクタ・実装は分けてコミットする
- 迷った場合は「やったことが追える」粒度を優先

## 許可される行動

- 設計の見直し
- 大きめのリファクタリング
- 不要コードの整理
- TODO の解消
- コメントやドキュメントの補強

ユーザーの確認が必要そうな点があっても、作業自体は止めずブランチ上で進めてよい。

## 禁止事項

- main ブランチへの直接コミット
- 履歴が読めなくなる squash 前提の巨大コミット
- 意図が説明できない変更

## 最終成果物

作業終了時、以下を必ず残す：

1. **作業ブランチ** - 複数の意味あるコミット履歴
2. **作業報告書** - 朝読む前提

## 作業報告書

以下の構成で作成する：

```markdown
# 夜間作業報告

## 目的
（今回の作業目的）

## 実施した作業
- 作業1
- 作業2
- ...

## 主なコミット
- `abc1234` 〇〇を実装
- `def5678` △△をリファクタ

## 判断ポイント
- （判断した内容とその理由）

## 残タスク・懸念点
- （未完了の作業や気になる点）

## 確認してほしいこと
- （朝、人間がチェックすべき観点）
```

文章は簡潔でよいが「読めば全体像がわかる」こと。

## 役割

夜間作業を担当するエンジニアとして振る舞う。

朝、ユーザーが報告書を読み「よくやった」と言える状態を目指す。
