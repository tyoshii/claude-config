# /ship [effort]

手元のコードを仕上げてから main へ送り出す一連の流れを自動で回す。
`/refine` → `/code-review` → `/pr-auto-merge`（内部で commit → push → PR 作成 →
auto-merge）を連続で起動し、CI をバックグラウンドで回しながら受け入れテストへ
スピード感もって移行する。

vibe coding 環境向け。実装が一段落したら `/ship` を打つだけで、品質チェック・
コミット・PR 作成・auto-merge 監視までを一気に進められる。

## 引数

- `effort` : `/code-review` のレビュー強度（`low` / `medium` / `high` / `max`）。
  省略時は `medium`。

## 前提

- 作業ブランチ上で、main へ送り出したい変更が手元にある状態で実行する
- `gh` CLI がインストール・認証済み
- リポジトリで auto-merge が有効化されている（`/pr-auto-merge` の前提）

## 実行手順

### 1. 開始前チェック

```bash
git branch --show-current
git status --short
```

送り出す変更があるかを確認する。**未コミットの変更（`git status --short` —
未追跡ファイル含む）**、または **リモートのデフォルトブランチに未反映のコミット**
のどちらかがあれば「送り出す変更あり」とみなす。後者は次で確認する（`origin/main`
は `git remote show origin` で得たデフォルトブランチ名に読み替える）：

```bash
git log --oneline origin/main..HEAD
```

- **未コミットの変更も未反映のコミットも一切ない場合** → 「送り出す変更が
  ありません」と報告して終了
- **メインブランチにいる場合**：
  - 判定は `git remote show origin | grep 'HEAD branch'` で得たデフォルト
    ブランチ名と、`git branch --show-current` の結果が一致するかで行う
  - そのまま進めると後続の `/pr-auto-merge`（内部の `/pr`）が作業ブランチを
    自動生成するので問題ないが、ユーザーに「メインブランチ上で実行中。
    `/pr` が作業ブランチを自動生成します」と一言伝えてから進む

### 2. /refine で仕上げ

`/refine` を実行し、変更コードの品質・効率・再利用性を改善する。

- `/refine` は指摘を自動で修正に反映する（このスキルの本来の挙動）
- 修正が入った場合、その内容を簡潔にユーザーへ報告する
- **`/refine` が `.claude/rules/` にルールファイルを追加・更新した場合**は、
  それが auto-merge 経由で main に混入する。一言ユーザーに知らせ、PR に含めて
  よいか確認する（不要なら除外する）
- **`/refine` が「スキップした指摘」（false positive・対応不要として除外した指摘）
  を報告した場合、その一覧を控えておく**。完了報告（ステップ 5）で番号つきで
  再掲する。スキップ指摘がなければ控えるものはない

### 3. /code-review で点検

`/code-review <effort>` を実行する（`effort` は引数。省略時 `medium`）。

`--fix` は付けない。`/code-review` の指摘は **bug findings（動作の正しさに関わる
バグ）** と **cleanup findings（再利用・簡素化・効率の改善）** の 2 系統で出る。
これに沿って扱いを分岐する：

- **cleanup findings のみ**（リファクタ・簡素化・効率の提案）→ Claude が自動で
  コードに反映し、反映内容を簡潔に報告する
- **bug findings が 1 件でもある場合**（データ破壊・セキュリティ・後方互換性の
  破壊など、動作の正しさに直接影響するもの）：
  - **そこで一旦止める**。bug findings の一覧をユーザーに提示し、
    「重大な指摘があります。修正してから送り出しますか？」と確認する
  - ユーザーが修正を指示 → Claude が指摘に沿って修正を反映し、ステップ 3 を
    再実行（再点検）。同じ指摘が 2〜3 周しても解消しない場合は止めて委ねる
  - ユーザーがこのまま進めると判断 → ステップ 4 へ
- 指摘がない、または cleanup findings の自動修正のみで済んだ場合 → ステップ 4 へ

bug か cleanup か判断に迷う指摘は bug 側に倒し、ユーザーに確認する。

### 4. /pr-auto-merge で送り出し

`/pr-auto-merge` を実行する。

- `/pr-auto-merge` は内部で `/pr merge` を呼び、未コミットの変更の **commit →
  push → PR 作成 → auto-merge 設定** までを一気に行う。したがって
  このスキルで別途 `/commit` を呼ぶ必要はない（commit はこの中で自動実行される）
- auto-merge が予約された場合、バックグラウンド監視エージェントが起動し、
  ユーザーはマージを待たずに次の作業（受け入れテスト等）へ移れる

### 5. 完了報告

```
## ship 完了

- refine: <修正あり（概要） / 変更なし>
- code-review (<effort>): <指摘なし / cleanup を自動修正（概要） / bug ありユーザー確認済み>
- PR: <PR URL>
- マージ: <即マージ完了 / auto-merge 予約・バックグラウンド監視中（完了時に通知）>

受け入れテストに移れます。

### refine がスキップした指摘

1. `<対象ファイル>` — <指摘内容>（スキップ理由: <理由>）
2. ...
```

- **「refine がスキップした指摘」セクション** には、ステップ 2 で控えた
  `/refine` のスキップ指摘を **番号つき** で列挙する。各項目は対象ファイル・
  指摘内容・スキップ理由を含める
- スキップ指摘が 1 件もなかった場合は、このセクションごと省略してよい
  （または「スキップした指摘はありません」と 1 行で書く）
- このセクションは「自動修正で握りつぶした判断を後から見直せるようにする」ためのもの。
  vibe coding でスピードを優先しつつ、見送った指摘を取りこぼさないための保険

- マージ行は `/pr-auto-merge` の戻りで確定させてから書く。即マージなら
  「即マージ完了」、auto-merge 予約なら「バックグラウンド監視中（完了時に通知）」。
  予約時はバックグラウンド監視エージェントの完了通知を受け取ったら、その結果を伝える

## 注意事項

- 各ステップは前のステップが完了してから次へ進む（並列実行しない）
- bug findings でユーザーが止めた場合、`/pr-auto-merge` までは進めない
- `/refine` `/code-review` が何も指摘しなくても、それ自体は正常。そのまま送り出す
- このスキルが直接 push・PR 操作を行うことはなく、すべて配下のスキルに委譲する

## 失敗時の扱い

いずれかのステップが失敗したら**その場で停止**し、(a) 完了したステップ・
(b) 失敗内容・(c) 現在の git / PR の状態を報告してユーザーに委ねる。勝手にリトライ
や巻き戻しはしない。特に：

- 前提（`gh` 認証・auto-merge 有効）が満たされず `/pr-auto-merge` が
  `gh pr merge --auto` でエラーになった場合 → auto-merge なしの即マージで進めるか、
  通常 PR 作成で止めるかをユーザーに確認する
- push がコンフリクト等で失敗した場合 → 状態を報告し、解決方法をユーザーに委ねる
