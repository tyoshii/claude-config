# /skill-feedback <スキル名>

現在の会話コンテキストをもとに、指定されたスキルへのフィードバックを生成する。

スキルを使った直後、または作業が一段落したタイミングで実行する。
会話の中で起きたこと（うまくいったこと、足りなかったこと、想定外だったこと）を分析し、スキルの改善案を提示する。

## 前提

- 現在の会話の中で、対象スキルを使った作業が行われていること
- 会話コンテキストに、実際のプロジェクトでの適用結果が含まれていること

## 実行手順

### 1. スキルファイルの読み込み

対象スキルの SKILL.md を読み込む：

```
skills/<スキル名>/SKILL.md
```

見つからない場合は `~/.claude/skills/` 配下も探す。

### 2. 会話コンテキストの分析

現在の会話を振り返り、以下の観点で情報を抽出する：

#### 2-1. スキルが想定していなかったこと

- スキルの手順にないが、実際には必要だったステップ
- スキルが前提としていたが、現実のプロジェクトでは成り立たなかった条件
- スキルの判定・分岐に含まれていなかったパターン

#### 2-2. うまくいったこと

- スキルの手順通りに進んで、期待通りの結果が得られた部分
- これは変えるべきでない。改善案で壊さないようにする

#### 2-3. ユーザーが手動で補正したこと

- スキルの指示とは異なる判断をユーザーがした箇所
- ユーザーが追加で指示を出した内容（スキルに含まれていれば不要だったもの）
- 「これも考慮して」「ここは違う」等のフィードバック

#### 2-4. 実際のプロジェクトから得られた知見

- 技術スタック固有の考慮事項（例: Vercel だとインメモリのレート制限が効かない）
- セキュリティ・データ境界の問題（例: 内部 API のレスポンスに個人情報が含まれていた）
- スケールや規模に関する想定とのズレ

### 3. フィードバックの整理

抽出した情報を以下の3カテゴリに分類する：

#### A: スキルに追加すべき内容

スキルの手順・チェックリスト・判定基準に追加すべき新しい要素。

- **どこに追加するか**（フェーズ、ステップ番号）を明示
- **なぜ必要か**（今回の経験から得られた理由）を添える
- 汎用的でないもの（特定プロジェクト固有の事情）は除外する

#### B: スキルを修正すべき内容

既存の手順・記述のうち、実態と合っていないもの。

- **現在の記述**と**修正案**を対比で示す
- 修正の理由を添える

#### C: 記録のみ（スキルには反映しない）

特定プロジェクト固有の知見で、スキルの汎用的な手順には含めないが、参考になるもの。

### 4. 改善案の提示

以下のフォーマットでユーザーに提示する：

```markdown
## スキルフィードバック: /<スキル名>

### 今回の適用サマリー
- プロジェクト: <概要>
- 使ったフェーズ: <どこまで進んだか>
- 所感: <一言>

### A: 追加提案

#### A-1. <追加内容のタイトル>
- **追加先**: フェーズ X / ステップ Y の後
- **理由**: <今回の経験>
- **内容**:
  <具体的な追加テキスト>

#### A-2. ...

### B: 修正提案

#### B-1. <修正内容のタイトル>
- **現在の記述**: <該当箇所の引用>
- **修正案**: <修正後のテキスト>
- **理由**: <なぜ変えるべきか>

#### B-2. ...

### C: プロジェクト固有の知見（参考）
- <知見1>
- <知見2>
```

### 5. クリップボードにコピー

スキルの管理リポジトリと作業中のリポジトリは別であることが多い。
提案書を直接スキルファイルに反映するのではなく、**クリップボードにコピー**してユーザーに渡す。

```bash
echo '<提案書全文>' | pbcopy  # macOS
```

コピー完了後、以下を案内する：

```
提案書をクリップボードにコピーしました。
スキル管理リポジトリで貼り付けて反映してください。
```

**このスキルはファイルの編集を行わない。**

## 注意事項

- 会話コンテキストがない状態（スキル未使用）で実行されたら、その旨を伝えて終了する
- 特定プロジェクト固有の事情を汎用スキルに混ぜない。判断に迷ったらカテゴリ C に分類する
- うまくいった部分を壊す修正は提案しない
- 大量の提案を出すより、インパクトの大きいものを 3-5 個に絞る
- スキルの根本的な方向性を変える提案（スコープの変更等）は、修正ではなく別途ユーザーと議論する
