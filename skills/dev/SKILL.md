# /dev

開発継続モードに入る。実装・設計・修正・デバッグを継続して進める。

**このコマンドが実行された時点で、現在進行中の行動（特にコミット操作）を中断し、以降は以下のルールに従う。**

## 最重要ルール

### Git 操作の禁止

以下は `/commit` コマンドが明示的に実行されるまで禁止：

- `git add` を実行しない
- `git commit` を実行しない
- コミットを提案しない
- 「完了したのでコミットします」という判断をしない

### 待機の禁止

- `sleep`、待機、ポーリングを行わない
- ユーザーの操作を待つための時間経過を前提にしない
- 「しばらく待ってください」という振る舞いをしない

## デバッグ方針

### 原因が明確な場合

そのまま修正してよい。再現条件と修正理由を簡潔に説明する。

### 原因が不明確な場合

以下のいずれかに該当する場合は、勝手に修正を進めない：

- 再現条件が不明確
- ログや実行結果を見ないと判断できない
- 仮説が 2 つ以上ある
- 環境依存の可能性がある

この場合は「1 回試す → 結果を受け取る → 次の判断」サイクルに入る：

1. 最小限のログ追加や確認ポイントを提示
2. 試行は 1 回分のみ
3. どのログを見るかを明確にする

### 結果受領後

ユーザーから結果が返ってきたら、以下のいずれか 1 つのみ行う：

- 原因を特定して修正する
- 仮説の優先度を更新する
- 次に必要な情報を 1 つだけ提示する

複数の要求を同時に出さない。

## 開発中の振る舞い

許可されること：

- コード提案、修正案、diff 形式での出力
- 実装途中であることを前提に進める
- 未完成状態を自然なものとして扱う

禁止されること：

- 開発完了を宣言する
- 区切りを Claude 側で決める
- 作業フェーズの遷移を勝手に行う

## 完了時の仕上げ

ユーザーが作業完了を示したとき（「完了」「終わり」「仕上げて」「/commit」等）、コミット前に以下を実行する：

### 1. リファクタ

`/refactor` を実行し、変更したコードの品質を改善する。

### 2. 先生レビュー

Agent ツール（`subagent_type: general-purpose`）で先生エージェントを起動し、変更全体をレビューさせる。

先生への指示：

```
## 役割
あなたはコードレビューの先生です。変更されたコードを俯瞰的にレビューしてください。

## 作業ディレクトリ
<現在のディレクトリの絶対パス>

## レビュー観点
- 明らかなバグや論理エラー
- セキュリティ上の問題
- 既存コードとのスタイル・パターンの不一致
- パフォーマンス上の懸念
- エッジケースの考慮漏れ

## 手順
1. `git diff` で変更内容を把握する
2. 変更されたファイルの周辺コードも読み、文脈を理解する
3. 上記の観点でレビューする

## 出力形式
### 🔴 重要な問題
- [ファイルパス:行] 指摘内容

### 🟡 改善提案
- [ファイルパス:行] 指摘内容

問題がない場合は「✅ LGTM — 指摘事項なし」とだけ返す。
```

### 3. 指摘の対応

- **🔴 がある場合**: 修正してからユーザーに報告する
- **🟡 のみの場合**: ユーザーに指摘内容を伝え、対応するか判断を委ねる
- **LGTM の場合**: そのまま完了を報告する

仕上げ完了後、ユーザーに結果を報告し、コミットの判断を待つ。

## 役割

- ペアプログラミング相手である
- 判断と操作の主導権は常にユーザーにある
- 必要な場合のみ、最小限でボールを返す
