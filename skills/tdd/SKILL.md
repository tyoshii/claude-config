# /tdd <実装したい機能の説明>

TDD（テスト駆動開発）で機能を実装する。Red → Green → Refactor サイクルを自律的に回し、最後に refine して結果を報告する。

## 前提

- パッケージ: `packages/web`（Vitest）または `packages/change-feed`（Vitest）
- テストファイルの配置: 対象ファイルと同階層の `__tests__/` ディレクトリに `<ファイル名>.test.ts` として作成
- テスト実行: `cd packages/web && npm test` または対象パッケージの `npm test`
- テストの書き方: 既存テスト（`packages/web/src/**/__tests__/*.test.ts`）のスタイルに従う
  - `describe` / `it` のラベルは日本語
  - `vi.mock` でモジュールをモック
  - `vi.mocked` で型安全にモック関数を扱う

## 実行手順

### 1. 設計フェーズ

タスク説明を分析し、実装に必要なテストケースを洗い出す。

1. 対象の既存コードを読み、現状の構造・依存関係を把握する
2. 実装すべき機能を小さな単位に分解する
3. 各単位に対してテストケースを列挙する：
   - 正常系（主要なパス）
   - 境界値・エッジケース
   - 異常系（エラーハンドリング）
4. テストケースの実装順序を決める（シンプルなものから複雑なものへ）

設計結果をユーザーに簡潔に共有する：

```
## TDD 設計

対象: <ファイルパス>
テストケース（実装順）:
1. <テストケース名> — <何を検証するか>
2. ...
```

**ユーザーの承認を待たず、そのまま実装に入る。**

### 2. TDD サイクル（自律実行）

以下のサイクルを全テストケースが完了するまで自律的に繰り返す。途中でユーザーに確認を求めない。

#### Red: テストを書く

- 1つのテストケースに対して、失敗するテストを書く
- テストを実行し、意図通り失敗することを確認する
- **テストが失敗する理由が「未実装だから」であることを確認する**（構文エラーや import エラーで失敗しているなら先に修正する）

#### Green: 最小限の実装

- テストを通すための最小限のコードを書く
- テストを実行し、パスすることを確認する
- 他のテストが壊れていないことも確認する（`npm test` で全テスト実行）

#### Refactor: リファクタリング

- テストが通った状態で、コードを改善する
  - 重複の排除
  - 命名の改善
  - 構造の整理
- リファクタリング後もテストがパスすることを確認する
- **不要なリファクタリングはしない。テストが通っていてコードが十分きれいならスキップする**

### 3. 全テスト確認

全サイクル完了後、`npm test` で全テストがパスすることを確認する。

### 4. Refine

`/refine` を実行し、実装コードとテストコードの両方を改善する。

### 5. 結果報告

以下の形式でユーザーに報告する：

```markdown
## TDD 完了

### 実装内容
- <何を実装したか（1〜3行）>

### テスト結果
- テストファイル: `<パス>`
- テストケース数: <N>件
- 結果: 全件パス ✅ / N件失敗 ❌

### テストケース一覧
1. ✅ <テストケース名>
2. ✅ <テストケース名>
...

### Refine 結果
<refine で行った改善内容をここに書く。何も改善がなかった場合は「改善なし」>

### 変更ファイル
- `<パス>` — <変更概要>
- `<パス>` — <変更概要>
```

**Refine 結果は必ず含める。** ユーザーにとって最も価値のある情報の1つ。

## ルール

### Git 操作の禁止

`/commit` コマンドが明示的に実行されるまで以下を禁止：
- `git add` / `git commit` を実行しない
- コミットを提案しない

### テスト実行の原則

- Red フェーズ: テストが失敗することを必ず確認する（いきなり Green にならないこと）
- Green フェーズ: 全テスト（`npm test`）を実行して既存テストが壊れていないことを確認する
- テスト実行でエラーが出たら、原因を特定して修正してから次に進む

### 実装の原則

- テストなしにコードを書かない
- テストを通すために必要な最小限のコードだけ書く
- 「あとで必要になりそう」な実装をしない（YAGNI）
- 既存のコードスタイル・パターンに合わせる
