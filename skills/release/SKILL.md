# /release

最新タグからの差分を確認し、リリースノート付きで新しいリリースを作成する。

## 実行手順

### 1. 前提確認

```bash
git branch --show-current
```

- メインブランチ（main）にいることを確認する
- メインブランチにいない場合は警告を出し、続行するか確認する

```bash
git fetch --tags
```

### 2. 最新タグと差分の取得

```bash
# 最新タグ
LATEST_TAG=$(git describe --tags --abbrev=0)

# 最新タグからの全コミット（マージコミット除外）
git log ${LATEST_TAG}..HEAD --oneline --no-merges

# コミット数
git rev-list ${LATEST_TAG}..HEAD --count --no-merges
```

- コミットが 0 件の場合は「リリースする差分がありません」と報告して終了

### 3. 次のバージョン番号の決定

最新タグのバージョンをインクリメントする：

- タグ形式: `v{major}.{minor}.{patch}`（例: `v3.0.56`）
- デフォルトは **patch** をインクリメント（例: `v3.0.56` → `v3.0.57`）
- 大きな機能追加がある場合は minor を上げることを提案してもよい

### 3.5. リリース前作業の自動検出

タグ作成前に、以下の2点を**必ず**チェックする。これはスキップできない必須チェックである。

#### A. マイグレーションの有無

```bash
git diff ${LATEST_TAG}..HEAD --name-only -- 'packages/admin/prisma/migrations/'
```

差分がある場合、リリース前に本番 DB へのマイグレーション適用が必要：
- `DATABASE_URL=<本番DB> npx prisma migrate deploy`（inu/neko 両方）

#### B. リリーススクリプトの有無

```bash
ls scripts/release/${新バージョン}/ 2>/dev/null
```

ディレクトリが存在しスクリプトがある場合、リリース前（またはデプロイ後）に実行が必要。
各スクリプトの先頭コメントに実行タイミング（デプロイ前/後）と対象（inu/neko/両方）を記載する前提。

#### C. メンテナンスモードの判断

以下のいずれかに該当する場合、メンテナンスモードを推奨する：
- マイグレーションにカラム削除・リネーム・型変更が含まれる（破壊的変更）
- リリーススクリプトがデータの大量更新を行う
- 認証やミドルウェアの変更が含まれる

該当しない場合（カラム追加のみ、コード変更のみ等）はメンテナンス不要。

#### 報告

チェック結果をリリースノートの後に以下の形式で表示する：

```
## リリース前作業

- マイグレーション: あり / なし
- リリーススクリプト: あり（scripts/release/v3.5.0/） / なし
- メンテナンスモード: 推奨 / 不要
```

**マイグレーションまたはリリーススクリプトがある場合、ユーザーに「リリース前作業を先に実行しますか？」と確認する。** 作業完了後にタグ push に進む。

### 4. リリースノートの作成

差分のコミットを分析し、以下の形式でリリースノートを作成する：

```markdown
## What's Changed

### Features
- 変更内容の説明 (#PR番号)

### Bug Fixes
- 修正内容の説明 (#PR番号)

### Others
- その他の変更内容 (#PR番号)
```

#### ルール

- コミットメッセージの言語（日本語 or 英語）に合わせる
- セクションは該当するコミットがある場合のみ表示する
- 各項目は **ユーザーにとって何が変わったか** を簡潔に書く（コミットメッセージの丸写しではなく、意味のある説明にする）
- PR 番号がコミットメッセージに含まれていれば記載する
- docs、chore、CI のみの変更は Others にまとめる
- 関連する複数コミット（同じ PR や同じ目的の修正）は 1 項目にまとめる

### 5. ユーザーに確認

以下を表示して確認を求める：

- 新しいバージョン番号
- リリースノートの内容
- 「このリリースノートで `{バージョン}` をリリースしますか？」

**ユーザーが明確に承認するまでリリースを実行しない。**

修正を求められた場合はリリースノートを修正して再度確認する。

### 6. タグの作成と push

```bash
git tag {新バージョン}
git push origin {新バージョン}
```

### 7. GitHub Release の作成

```bash
gh release create {新バージョン} --title "{新バージョン}" --notes "{リリースノート}"
```

- `--notes` にはステップ 4 で作成したリリースノートを HEREDOC で渡す

### 8. GitHub Actions の確認

タグ push により production 向けの GitHub Actions ワークフローが起動する。全ワークフローの完了を確認する。

#### 8-1. ワークフローの起動を待つ

タグ push 直後はワークフローがまだ起動していない場合がある。数秒待ってから確認を開始する。

#### 8-2. 完了まで監視する

```bash
gh run list --branch {新バージョン} --limit 30 --json status,conclusion,name --jq '.[] | "\(.conclusion // .status)\t\(.name)"'
```

- `in_progress` や `queued` のワークフローがなくなるまで繰り返し確認する（30秒〜1分間隔）
- 確認は最大 10 分程度を目安とする。それ以上かかる場合はユーザーに状況を報告して判断を仰ぐ

#### 8-3. 結果の集計と報告

全ワークフロー完了後、成功・失敗を集計してユーザーに報告する：

```
✅ 成功: 12 ワークフロー
❌ 失敗: 2 ワークフロー
  - inu Web Preview Production CI/CD
  - neko Web Preview Prod CI/CD
```

**失敗がある場合**、エラーログを取得して原因を報告する：

```bash
# 失敗した run の ID を取得
gh run list --branch {新バージョン} --limit 30 --json conclusion,databaseId,name --jq '.[] | select(.conclusion == "failure") | "\(.databaseId)\t\(.name)"'

# エラーログを取得
gh run view {run_id} --log-failed
```

エラーの内容と影響範囲をユーザーに伝え、対応が必要かどうかを判断できるようにする。

### 9. 結果の報告

- リリースの URL を表示する
- GitHub Actions の結果サマリーを報告する
- 失敗がある場合はエラー内容と影響範囲を報告する
- 「Terraform Cloud の production ワークスペースで Plan が発行されます。Apply は手動で実行してください。」と案内する（タグ push で Terraform Cloud がトリガーされるため）

## 注意事項

- `gh` CLI がインストール・認証済みであることが前提
- リリースノートの確認前にタグを作成しない
- production への Terraform Apply は自動実行されない（Plan のみ）ため、必ず案内する
