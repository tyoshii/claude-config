# /test-strategy

プロジェクトのコードベースを分析し、テスト戦略を提案する。

「何をどうテストすべきか」「PR の品質ゲートをどう構成するか」「Preview 環境は実現できるか」を調査し、実装ロードマップとして提示する。

テストの実装自体はこのスキルのスコープ外。提案を見てユーザーが判断し、`/teamdev` や `/overnight` で実装する。

## 実行手順

### 1. コードベース調査

Agent ツール（`subagent_type: Explore`）を**並列に**使い、以下を把握する。

#### 1-1. 技術スタック

- 言語・フレームワーク・ディレクトリ構造
- フロントエンド / バックエンド / モバイル / モノレポ構成
- ビルドツール（webpack, vite, turbo, etc.）
- デプロイ先（Vercel, AWS, GCP, Docker, etc.）

#### 1-2. 既存テストの棚卸し

- テストフレームワークの有無（Jest, Vitest, pytest, RSpec, Go test, etc.）
- テストファイルの数と分布
- テストの種類（unit / integration / e2e）の比率
- テストカバレッジの計測有無
- テストの実行方法（`npm test`, `make test`, etc.）

#### 1-3. CI/CD の現状

- CI ツール（GitHub Actions, CircleCI, GitLab CI, etc.）
- 既存のワークフロー / パイプラインの内容
- テストが CI で実行されているか
- PR へのステータスチェック・コメント投稿の有無

#### 1-4. デプロイ・環境構成

- Preview 環境の有無（Vercel Preview, Netlify Deploy Preview, etc.）
- Docker / docker-compose の有無
- 環境変数・シークレットの管理方法
- データベース・外部サービスへの依存

#### 1-5. アプリケーションの特性

- 主要なユーザーフロー（認証、CRUD、決済、etc.）
- API エンドポイントの数と構造
- フロントエンドのページ数・ルーティング構造
- データの機密性（個人情報、決済情報 etc.）

### 2. テスト戦略の策定

調査結果をもとに、以下の観点でテスト戦略を策定する。

#### 2-1. テストピラミッドの設計

プロジェクトの特性に合わせたテスト配分を提案する：

```
    /  E2E  \        ← 少数・重要フローのみ
   / Integration \   ← API・DB連携の検証
  /     Unit       \ ← ロジック・ユーティリティ
```

各層で何をテストすべきかを具体的に列挙する：

- **Unit** — どのモジュール / 関数をテストすべきか
- **Integration** — どの連携ポイント（API↔DB、サービス間 etc.）をテストすべきか
- **E2E** — どのユーザーフローをテストすべきか（ログイン、主要 CRUD、決済 etc.）

#### 2-2. テストフレームワークの選定

技術スタックに合ったフレームワークを提案する：

| レイヤー | 候補 | 選定基準 |
|----------|------|----------|
| **Unit / Integration** | 既存フレームワークを優先。なければ言語の標準的なもの | 学習コスト、エコシステムとの相性 |
| **E2E（Web）** | Playwright / Cypress | Playwright を優先（マルチブラウザ、CI 親和性）。既に Cypress が入っていればそちらを継続 |
| **E2E（API）** | supertest / httpx / 言語標準 | フレームワークのテストクライアントがあればそれを使う |
| **Visual Regression** | Playwright screenshots / Percy / Chromatic | コスト・既存ツールとの相性で判断 |

#### 2-3. テスト対象の優先順位

すべてを一度にテストするのは現実的でない。以下の基準で優先順位をつける：

1. **壊れたら致命的なもの** — 認証、決済、データ整合性
2. **変更頻度が高いもの** — git log から変更頻度を分析
3. **手動テストのコストが高いもの** — 複数ステップのフロー、条件分岐が多い処理
4. **過去にバグが出たもの** — issue / commit ログから推測

### 3. PR 品質ゲートの設計

PR を出したときに自動で品質を検証する仕組みを設計する。

#### 3-1. diff → 影響範囲の特定

PR の変更内容から、影響を受ける機能を特定する方法を設計する：

- **ファイルパスベース** — `src/auth/` の変更 → 認証関連テストを実行
- **依存グラフベース** — import 解析で変更ファイルに依存するモジュールを特定
- **タグ / ラベルベース** — テストに `@auth`, `@payment` 等のタグをつけ、変更に応じたテストを選択実行

プロジェクトの規模に応じて、現実的な方法を提案する。小規模なら全テスト実行で十分。

#### 3-2. CI ワークフローの設計

```
PR push
  ↓
[Lint / Type Check]  ← 即時実行（軽い）
  ↓
[Unit / Integration Tests]  ← 並列実行
  ↓
[E2E Tests]  ← 必要に応じて（重い）
  ↓
[結果を PR コメントに投稿]
```

提案する内容：
- ワークフローファイルの構成（`.github/workflows/` 等）
- テスト実行の並列化戦略
- キャッシュ戦略（依存関係、ブラウザバイナリ等）
- 実行時間の目安と許容範囲

#### 3-3. PR コメントへの結果掲示

テスト結果を PR に自動投稿する方法を設計する：

- **ステータスチェック** — pass/fail を GitHub のチェックとして表示
- **コメント投稿** — テスト結果のサマリー、カバレッジ差分、スクリーンショット等
- **失敗時の情報** — どのテストが落ちたか、エラーメッセージ、再現手順

### 4. Preview 環境の実現可能性

#### 4-1. 判定

| デプロイ先 | Preview 環境の実現方法 | 難易度 |
|------------|----------------------|--------|
| **Vercel** | 自動（PR ごとに Preview URL が生成される） | ほぼゼロ |
| **Netlify** | 自動（Deploy Preview） | ほぼゼロ |
| **Cloudflare Pages** | 自動（Preview Deployment） | ほぼゼロ |
| **AWS / GCP + Docker** | docker-compose で一時環境を構築、または ECS/Cloud Run にデプロイ | 中〜高 |
| **オンプレ / VM** | Docker 必須。リソースの確保が課題 | 高 |
| **DB 依存が重い** | テスト用 DB のシード / マイグレーションが必要 | 中〜高 |

#### 4-2. 提案内容

- Preview 環境の構成案（実現可能な場合）
- PR コメントへの Preview URL 自動投稿
- 環境のライフサイクル（PR close 時に自動削除）
- 実現が難しい場合の代替案（E2E テスト結果 + スクリーンショット）

### 5. 実装ロードマップ

すべての提案を優先度順に並べたロードマップを作成する。

#### フォーマット

```markdown
## テスト戦略 実装ロードマップ

### 現状サマリー
- テストの有無: <あり / なし / 部分的>
- CI の有無: <あり / なし>
- Preview 環境: <あり / なし / 自動>
- 主なリスク: <テストがない領域、壊れやすい領域>

### Phase 1: 基盤整備（最優先）
- [ ] <テストフレームワークのセットアップ>
- [ ] <CI ワークフローの作成>
- [ ] <最初のテスト N 件>
想定工数: /overnight 1-2 回分

### Phase 2: 主要フローのカバー
- [ ] <認証フローの E2E テスト>
- [ ] <主要 CRUD の Integration テスト>
- [ ] <PR コメントへの結果投稿>
想定工数: /teamdev 1-2 回分

### Phase 3: 品質ゲートの強化
- [ ] <カバレッジ計測・閾値設定>
- [ ] <Visual Regression（必要なら）>
- [ ] <影響範囲ベースのテスト選択実行>
想定工数: /overnight 1-2 回分

### Phase 4: Preview 環境（実現可能な場合）
- [ ] <Preview 環境の構築>
- [ ] <PR への URL 自動投稿>
- [ ] <環境の自動クリーンアップ>
想定工数: <構成による>

### 推奨する実行順序
Phase 1 → Phase 2 → Phase 3 → Phase 4
各フェーズは独立しており、途中で止めても価値がある。
```

## 注意事項

- このスキルは**調査と提案のみ**。ファイルの作成・変更は行わない
- 提案は具体的に。「テストを書きましょう」ではなく「この関数のこのケースをテストすべき」レベルで
- フレームワークやツールの提案には、なぜそれを選んだかの理由を添える
- 実装の想定工数は `/overnight` や `/teamdev` の回数で表現する
- プロジェクトの規模に見合わない過剰な提案はしない（個人開発に Percy は不要）
