# /api-stack <タスク説明>

既存システムに Public API を実装する。
生成 AI との連携を前提に、プログラマブルなインターフェースをシステムに追加する。

CLI や MCP は公開された API のインターフェースにすぎないため、このスキルのスコープ外とする。

## 概要

多くの既存システムには API が外出しされていない。
このスキルはコードベースを解析し、以下を実装する：

- **Public API** — 既存の内部 API を公開、または新規に実装
- **認証** — API トークンの発行・管理
- **レート制限** — トークン単位のスロットリング
- **ドキュメント** — API リファレンスの整備

## 実行手順

### 0. ブランチ運用

作業は専用ブランチで行う。

```
api-stack/yyyy-mm-dd-<topic>
```

例: `api-stack/2026-03-20-user-management`

**禁止**: main / develop など既存ブランチへの直接コミット

コミットは意味のある単位で随時行う。`/commit` の手順に従う。

### 1. フェーズ 1 — コードベース調査

Agent ツール（`subagent_type: Explore`）を**並列に**使い、以下を把握する。

#### 1-1. アーキテクチャ全体像

- 言語・フレームワーク・ディレクトリ構造
- エントリポイント（Web サーバー、ルーター、コントローラー）
- データベース・外部サービスとの接続方式
- 認証・認可の仕組み（既存のものがあるか）
- `.claude/rules/` があれば読み込む

#### 1-2. API の有無と状態

以下の観点で既存 API を探す：

- **ルーティング定義** — Express router、Rails routes、FastAPI path、etc.
- **コントローラー / ハンドラー** — HTTP リクエストを処理するレイヤー
- **シリアライザー / レスポンス定義** — JSON レスポンスの構造
- **API ドキュメント** — OpenAPI/Swagger、API Blueprint、etc.
- **内部 API** — フロントエンドが呼ぶ REST/GraphQL エンドポイント
- **RPC / gRPC** — 内部サービス間通信

#### 1-3. セキュリティ・データ境界の調査

内部 API をそのまま公開すると事故になるケースが多い。以下を重点的に調査する：

- **アクセス制御の前提** — 内部 API が「ログインユーザー＝本人」を前提にしていないか。セッション Cookie ベースの認証がトークン認証に置き換わったとき、アクセス制御が崩れないか
- **データアクセス境界** — マルチテナント / チーム / ユーザー間のデータ分離がどの層で実現されているか（DB の RLS、アプリ層のミドルウェア、コントローラーの where 句 etc.）。API トークンに紐づくユーザーが他者のデータにアクセスできないか
- **レスポンスの情報量** — 内部 API のレスポンスに、UI 側で表示制御している機密フィールド（個人情報、内部 ID、管理者フラグ等）が含まれていないか。API だと JSON が丸ごと返るため、フロントの表示制御は防壁にならない
- **破壊的操作の洗い出し** — DELETE、bulk 操作（一括更新・一括削除）、管理者専用操作を一覧化する。外部に公開するかの判断材料にする
- **DB アクセスの権限** — サービスロールキー（RLS バイパス）やスーパーユーザー接続を使っていないか。アプリ層のアクセス制御が唯一の防壁になっている場合、テストを厚くする必要がある

#### 1-4. 判定結果

調査結果を以下のいずれかに分類する：

| 状態 | 説明 | 次のアクション |
|------|------|----------------|
| **A: 公開可能な API がある** | 内部 API が整備されており、認証を追加すれば外部公開できる | → フェーズ 2A |
| **B: 内部 API はあるが公開に不十分** | エンドポイントはあるが、レスポンス形式・エラーハンドリング・ドキュメントが不足 | → フェーズ 2B |
| **C: API が存在しない** | ビジネスロジックが UI に密結合しており、API レイヤーがない | → フェーズ 2C |

調査結果とフェーズ判定をユーザーに報告し、方針を確認する。

### 2. フェーズ 2 — API の実装 / 公開

#### 公開範囲の原則: 段階的公開

**内部 API をそのまま外部公開してはいけない。** 内部 API はフロントエンドが表示制御する前提で設計されており、レスポンスに不要な情報が含まれていたり、アクセス制御がフロント依存だったりする。

公開は以下の順で段階的に進める：

1. **まず読み取り専用（GET）のみ** — データ漏洩がないことを確認
2. **次に作成・更新（POST/PUT/PATCH）** — バリデーションとアクセス制御を確認
3. **最後に削除・bulk 操作** — 破壊的操作は慎重に。不要なら公開しない

各エンドポイントについて「公開する / 公開しない / 将来検討」を判断し、ユーザーに確認する。

#### バージョニング

外部公開 API には必ずバージョンプレフィックスをつける：

```
/api/v1/users
/api/v1/orders
```

- 内部 API はフロントと同時デプロイなのでバージョン不要だが、外部 API は後方互換性の破壊がクライアントを壊す
- v1 から始め、破壊的変更が必要になったら v2 を並行稼働する
- 既存の内部 API パスとは別のパスに公開 API をマウントする（内部: `/api/xxx`, 公開: `/api/v1/xxx`）

#### レスポンスフィールドの制限

内部 API のレスポンスをそのまま返さない。公開用のレスポンス定義を別途作成する：

- **除外すべきフィールド**: 内部 ID（auto increment）、管理者フラグ、内部ステータス、他ユーザーの個人情報、DB のカラム名がそのまま露出しているもの
- **実装方法**: シリアライザー / DTO / レスポンス変換層を設ける。内部モデルを直接 JSON 化しない
- **判断基準**: 「このフィールドが API ドキュメントに載っていて違和感がないか」

#### フェーズ 2A: 既存 API の公開

既に整備された内部 API がある場合。

1. **エンドポイントの棚卸し** — 公開対象のエンドポイントを一覧化（公開する / しない / 将来検討に分類）
2. **レスポンスフィールドの制限** — 公開用のレスポンス定義を作成
3. **アクセス制御の検証** — トークンに紐づくユーザーが、自分のデータにしかアクセスできないことを確認
4. **認証レイヤーの追加** — ステップ 2-Auth 参照
5. **レート制限の追加** — ステップ 2-Rate 参照
6. **ドキュメントの整備** — ステップ 2-Docs 参照

#### フェーズ 2B: 内部 API の改善 + 公開

内部 API はあるが公開品質に達していない場合。

1. **レスポンス形式の統一** — 一貫した JSON 構造に整理
   ```json
   {
     "data": { ... },
     "meta": { "page": 1, "total": 100 }
   }
   ```
   エラーレスポンス：
   ```json
   {
     "error": {
       "code": "VALIDATION_ERROR",
       "message": "...",
       "details": [...]
     }
   }
   ```
2. **レスポンスフィールドの制限** — 公開用のレスポンス定義を作成
3. **エラーハンドリングの統一** — HTTP ステータスコードの適切な使い分け
4. **バリデーションの追加** — リクエストパラメータの検証
5. **アクセス制御の検証** — トークンに紐づくユーザーが、自分のデータにしかアクセスできないことを確認
6. **認証レイヤーの追加** — ステップ 2-Auth 参照
7. **レート制限の追加** — ステップ 2-Rate 参照
8. **ドキュメントの整備** — ステップ 2-Docs 参照

#### フェーズ 2C: API の新規実装

API レイヤーが存在しない場合。

1. **ビジネスロジックの特定** — UI に埋まっているロジックを抽出
2. **API 設計** — RESTful エンドポイントの設計
   - リソース指向の URL 設計（`/api/v1/users`, `/api/v1/orders`）
   - 適切な HTTP メソッドの使い分け
   - ページネーション、フィルタリング、ソートの設計
   - 公開するエンドポイントの範囲を決定（まず読み取りから）
3. **サービスレイヤーの抽出** — UI → ビジネスロジック → データアクセスの分離
4. **レスポンス定義の作成** — 公開に適したフィールドのみ含むレスポンス型を設計
5. **API エンドポイントの実装** — フレームワークの慣習に従う
6. **アクセス制御の実装** — リソースの所有者チェック、テナント分離
7. **認証レイヤーの追加** — ステップ 2-Auth 参照
8. **レート制限の追加** — ステップ 2-Rate 参照
9. **ドキュメントの整備** — ステップ 2-Docs 参照

### 2-Auth: 認証 — API トークンの実装

#### トークン設計

- **形式**: ランダム文字列のベアラートークン（`sk-xxxxxxxxxxxx` 形式など、プロジェクトに適した prefix を選択）
- **保存**: ハッシュ化して DB に保存（平文保存は禁止）
- **スコープ**: 必要に応じて読み取り専用 / 読み書き等のスコープを設定

#### トークン管理 UI

プロダクトの既存 UI に合わせてトークン管理画面を実装する：

- **管理画面がある場合** → 管理画面内にトークン管理セクションを追加
- **マイページがある場合** → マイページ内に「API キー」セクションを追加
- **どちらもない場合** → 最小限のトークン管理エンドポイントを API として実装

管理画面で可能にする操作：
- トークンの発行（発行時のみ平文を表示、以降は末尾数文字のみ表示）
- トークンの一覧表示（名前、作成日、最終使用日、スコープ）
- トークンの無効化（削除ではなく無効化で履歴を残す）

#### 認証ミドルウェア

```
Authorization: Bearer sk-xxxxxxxxxxxx
```

- トークン未指定 → `401 Unauthorized`
- 無効なトークン → `401 Unauthorized`
- スコープ不足 → `403 Forbidden`
- 認証成功 → リクエストにユーザー情報を付与して次のミドルウェアへ

### 2-Rate: レート制限

#### 実装方針

レート制限はインフラ構成に依存するため、以下の順で検討する：

1. **リバースプロキシ（nginx/Cloudflare 等）で制御** — 既に使われていればそこで設定
2. **アプリケーションミドルウェアで制御** — Redis が使えれば Token Bucket / Sliding Window
3. **インメモリで制御** — 単一プロセスなら簡易的なインメモリカウンター（スケールしないが最小構成）

#### トークン単位の制限

- デフォルト: 60 req/min（調整可能）
- トークンごとに個別のレート制限を設定可能にする
- レスポンスヘッダーで残量を通知：
  ```
  X-RateLimit-Limit: 60
  X-RateLimit-Remaining: 45
  X-RateLimit-Reset: 1679000000
  ```
- 制限超過時: `429 Too Many Requests` + `Retry-After` ヘッダー

#### 制限の管理

- トークン管理 UI から個別のレート制限を設定できるようにする
- デフォルト値はアプリケーション設定で管理

### 2-Docs: API ドキュメント

#### ドキュメント形式の選択

コードベースの技術スタックに合わせて最適な方式を選択する：

| 方式 | 適用条件 | ツール |
|------|----------|--------|
| **コード生成型** | フレームワークがサポートしている場合 | FastAPI の自動生成、Swagger annotations、etc. |
| **OpenAPI spec ファイル** | 上記が使えない場合 | `openapi.yaml` を手書き + Swagger UI / Redoc で表示 |
| **マークダウン** | 最小構成の場合 | `docs/api.md` |

#### ドキュメントの内容

各エンドポイントについて以下を記載する：

- **URL とメソッド**
- **認証** — 必要なスコープ
- **リクエスト** — パラメータ、ボディの型と説明
- **レスポンス** — 成功時・エラー時のレスポンス例
- **レート制限** — 適用される制限

#### ドキュメントの配置

- **コード生成型**: フレームワークのルートに自動マウント（例: `/api/docs`）
- **OpenAPI spec**: `/api/docs` に Swagger UI / Redoc をマウント
- **マークダウン**: リポジトリの `docs/api.md` に配置

#### Getting Started セクション

ドキュメントの冒頭に以下を含める：

1. API トークンの取得方法
2. 最初のリクエスト例（curl）
3. 認証ヘッダーの指定方法
4. レート制限の説明

### 3. フェーズ 3 — 統合・仕上げ

#### 3-1. テスト

- API エンドポイントのテスト（正常系・異常系・認証・レート制限）
- **アクセス制御テスト** — ユーザー A のトークンでユーザー B のデータにアクセスできないこと
- **レスポンスフィールドテスト** — 公開 API のレスポンスに内部専用フィールドが含まれていないこと
- **破壊的操作の保護テスト** — スコープ不足のトークンで DELETE/bulk 操作が拒否されること

#### 3-2. PR 作成

`/pr draft review` の手順に従って PR を作成する。

## 規模に応じた実行戦略

### 小規模（API エンドポイント 5 個以下）

順次実行。1 つのエージェントで通しで実装する。

### 中規模（API エンドポイント 6-15 個）

teamdev 式に分散する：

```
[Explore x2] 調査
  ↓
[Agent A] API 実装 + 認証
[Agent B] API ドキュメント + レート制限
[先輩] 中間レビュー
  ↓ 統合・テスト・PR
```

### 大規模（API エンドポイント 16 個以上）

フェーズごとにブランチを分けて段階的に PR を出す：

1. `api-stack/api-core` — API エンドポイント実装 → PR & マージ
2. `api-stack/api-auth` — 認証 + レート制限 → PR & マージ
3. `api-stack/api-docs` — ドキュメント整備 → PR & マージ

各フェーズ内で teamdev 式に分散する。

## コミット方針

- 作業途中でも、意味のある単位でコミットする
- `/commit` の手順に従う
- 以下のように分けてコミットする：
  - 調査・設計メモ
  - API 実装（エンドポイントごと or 機能ごと）

## 禁止事項

- main ブランチへの直接コミット
- 既存機能の破壊的変更（API 追加は既存に影響しない形で）
- API トークンやシークレットのハードコード
- テストなしでの PR 作成
