# /refactor

変更されたコードをレビューし、品質・効率・再利用性を改善する。
ビルトインの `/simplify` と同等の改善に加え、プロジェクトルールの参照と更新を行う。

## フェーズ 1 — ルールの読み込み

1. `.claude/rules/` ディレクトリが存在するか確認する
2. 存在する場合、すべてのルールファイルを読み込み、内容を把握する
3. 読み込んだルールを以降のレビュー基準に加える

## フェーズ 2 — 変更ファイルの特定

1. `git diff HEAD` と `git diff --cached` で変更されたファイルを特定する
2. 変更がない場合は `git diff HEAD~1` で直近コミットの変更を対象にする
3. 対象ファイルの現在の内容を読む

## フェーズ 3 — 3 並列レビュー

以下の 3 つの観点で **並列にエージェントを起動** してレビューする：

### Agent 1: コード再利用
- 重複コードの検出
- 既存ユーティリティ・ヘルパーとの統合可能性
- 共通パターンの抽出候補

### Agent 2: コード品質
- 命名の一貫性
- エラーハンドリングの適切さ
- 型安全性
- 既存ルールへの準拠

### Agent 3: 効率・パフォーマンス
- 不要な計算・アロケーション
- N+1 クエリなどのパフォーマンス問題
- インポートの整理

$ARGUMENTS が指定されている場合は、その観点を全エージェントの重点項目に追加する。

## フェーズ 4 — 修正の適用

1. 3 つのレビュー結果を統合する
2. 指摘事項を重要度順にソートする
3. 修正を適用する
4. テスト・ビルドがあれば実行し、壊れていないことを確認する

## フェーズ 5 — ルールの更新

リファクタリングを通じて以下に該当する **新しいパターンやルール** を発見した場合、`.claude/rules/` に追加する：

- プロジェクト全体に適用できる一般的なコーディング規約
- 繰り返し現れるアンチパターンとその修正方針
- フレームワーク・ライブラリ固有のベストプラクティス

### ルール追加の基準

以下の **すべて** を満たす場合のみ追加する：

1. 今回の変更に限定されない、プロジェクト全体に適用できるルールである
2. 既存のルールと重複しない
3. コードから自明でない（コードを読めばわかることは書かない）

### ルールファイルのフォーマット

```markdown
# ルールタイトル

ルールの内容を簡潔に記述する。

## 理由

なぜこのルールが必要かを説明する。

## 良い例

良いコード例を示す。

## 悪い例

避けるべきコード例を示す。
```

### 更新結果の報告

- 新しく追加したルールがあれば、ファイル名と概要を報告する
- 追加すべきルールがなかった場合はその旨を報告する

## 出力

最後に以下をまとめて報告する：

1. **改善サマリ** — 実施した変更の一覧（ファイル名と内容）
2. **ルール更新** — 追加・更新したルールの一覧（なければ「なし」）
