# /finalize [effort]

チーム開発で実装完了後に品質チェック→改善→最終チェック→PR 作成を一連で行う。
`/code-review` → `/refine` → `/code-review`（最終）→ `/pr` の流れを自動で回し、
最終レビュー結果を PR コメントに記載する。

致命的な問題が見つかった場合は修正→refine→code-review をループし、
品質が確保されてから PR を作成する。

## 引数

- `effort` : `/code-review` のレビュー強度（`low` / `medium` / `high` / `max`）。
  省略時は `medium`。

## 前提

- 作業ブランチ上で、実装が完了した変更が手元にある状態で実行する
- `gh` CLI がインストール・認証済み

## 実行手順

### 1. 開始前チェック

```bash
git branch --show-current
git status --short
```

送り出す変更があるかを確認する。**未コミットの変更（`git status --short` —
未追跡ファイル含む）**、または **リモートのデフォルトブランチに未反映のコミット**
のどちらかがあれば「送り出す変更あり」とみなす。後者は次で確認する（`origin/main`
は `git remote show origin` で得たデフォルトブランチ名に読み替える）：

```bash
git log --oneline origin/main..HEAD
```

- **未コミットの変更も未反映のコミットも一切ない場合** → 「送り出す変更が
  ありません」と報告して終了
- **メインブランチにいる場合**：
  - 判定は `git remote show origin | grep 'HEAD branch'` で得たデフォルト
    ブランチ名と、`git branch --show-current` の結果が一致するかで行う
  - そのまま進めると後続の `/pr` が作業ブランチを自動生成するので問題ないが、
    ユーザーに「メインブランチ上で実行中。`/pr` が作業ブランチを自動生成します」
    と一言伝えてから進む

### 2. code-review（1回目）— 初期点検

`/code-review <effort>` を実行する（`effort` は引数。省略時 `medium`）。

`--fix` は付けない。`/code-review` の指摘は **bug findings（動作の正しさに関わる
バグ）** と **cleanup findings（再利用・簡素化・効率の改善）** の 2 系統で出る。
これに沿って扱いを分岐する：

- **cleanup findings のみ** → Claude が自動でコードに反映し、反映内容を簡潔に
  報告する。ステップ 3 へ
- **bug findings が 1 件でもある場合**：
  - bug findings の一覧をユーザーに提示し、Claude が指摘に沿って修正を反映する
  - 修正内容を簡潔に報告し、ステップ 3 へ
- **指摘なし** → そのままステップ 3 へ

### 3. /refine で品質改善

`/refine` を実行し、変更コードの品質・効率・再利用性を改善する。

- `/refine` は指摘を自動で修正に反映する
- 修正が入った場合、その内容を簡潔にユーザーへ報告する
- `/refine` が `.claude/rules/` にルールファイルを追加・更新した場合は、
  ユーザーに一言知らせる
- `/refine` がスキップした指摘があれば控えておく（完了報告で再掲する）

### 4. code-review（2回目）— 最終チェック

再度 `/code-review <effort>` を実行する。

**ここがこのスキルの核心。** refine 後の最終品質を確認する。

結果の扱い：

- **指摘なし、または cleanup findings のみ**：
  - cleanup findings があれば自動修正する
  - **レビュー結果をそのまま控えておく**（ステップ 6 で PR コメントに記載する）
  - ステップ 5 へ
- **bug findings が 1 件でもある場合**（致命的な問題）：
  - bug findings の一覧をユーザーに提示する
  - Claude が指摘に沿って修正を反映する
  - **ステップ 3 に戻る**（refine → code-review のループ）
  - 同じループが **3 回** に達しても bug findings が残る場合は止めて、
    残りの指摘をユーザーに報告し判断を委ねる。ユーザーがこのまま進めると
    判断した場合はステップ 5 へ

### 5. /pr で PR 作成

`/pr` を実行する。PR を作成する（commit → push → PR 作成）。

`/pr` の手順に従い、Draft PR ではなく通常の PR を作成する。

### 6. 最終レビュー結果を PR コメントに記載

ステップ 4 の最終 code-review の結果を PR コメントとして投稿する。

```bash
gh pr comment <PR番号> --body "<レビューコメント>"
```

コメントの形式：

```markdown
## Finalize — 最終コードレビュー結果

**レビュー強度:** <effort>

### 結果
<指摘なし: ✅ 問題なし / 指摘あり: 指摘の要約>

### 詳細
<code-review の指摘内容をそのまま記載。cleanup の自動修正内容も含める>

### refine サマリ
<refine で行った改善の要約>

---
🤖 `/finalize` による自動レビュー
```

- 指摘がゼロだった場合も「✅ 問題なし」として投稿する
- ループした場合は、何回ループしたかと最終的な状態を記載する
- PR の body ではなく、PR コメント（comment）に記載する

### 7. 完了報告

```
## finalize 完了

- code-review (1回目, <effort>): <指摘なし / cleanup 自動修正 / bug あり修正済み>
- refine: <修正あり（概要） / 変更なし>
- code-review (2回目, <effort>): <指摘なし / cleanup のみ / bug あり → ループN回>
- PR: <PR URL>
- レビューコメント: PR に投稿済み

### refine がスキップした指摘

1. `<対象ファイル>` — <指摘内容>（スキップ理由: <理由>）
2. ...
```

- スキップ指摘が 1 件もなかった場合はこのセクションごと省略

## 注意事項

- 各ステップは前のステップが完了してから次へ進む（並列実行しない）
- bug findings でループが発生した場合、最大 3 回まで
- このスキルが直接 push・PR 操作を行うことはなく、すべて配下のスキルに委譲する
- `/ship` との違い: `/ship` は vibe coding 向けで auto-merge まで行う。
  `/finalize` はチーム開発向けで、2 回目の code-review 結果を PR コメントに
  残し、チームメンバーがレビューしやすくする

## 失敗時の扱い

いずれかのステップが失敗したら**その場で停止**し、(a) 完了したステップ・
(b) 失敗内容・(c) 現在の git / PR の状態を報告してユーザーに委ねる。勝手にリトライ
や巻き戻しはしない。
