# /teamdev <タスク説明>

エージェントチームを編成して自律的に開発を進める。

ユーザーはタスクを投げたら待つだけ。
エージェントたちが調査・設計・実装・テストを分担して進める。

## 実行手順

### 1. タスクの分解

タスク説明を分析し、並列に進められるサブタスクに分解する。

分解の方針：

- **依存関係がないもの**は並列に進める
- **依存関係があるもの**はフェーズを分けて順次実行
- 各サブタスクは 1 エージェントが完結できる粒度にする

### 2. .claude/rules の確認

作業に入る前に、`.claude/rules/` ディレクトリを確認する。

1. `.claude/rules/` 配下のファイルを Glob で一覧する
2. 各ルールファイルを読み、内容を把握する
3. 以降のすべてのフェーズで、ルールに従って作業を進める

ルールファイルが存在しない場合はスキップして次に進む。

### 3. フェーズ 1 — 調査

まず Agent ツール（`subagent_type: Explore`）を**並列に**使い、以下を把握する：

- 既存コードの構造と関連ファイル
- 使われている技術スタック・パターン
- テストの有無と実行方法
- CLAUDE.md やプロジェクト固有のルール

調査結果をもとに、実装計画を組み立てる。

### 4. フェーズ 2 — 実装

Agent ツール（`subagent_type: general-purpose`）を使い、サブタスクごとにエージェントを起動する。
**同時に先輩エージェントも起動する。**

#### 実装エージェントへの指示テンプレート

各実装エージェントには以下を伝える：

```
## 作業ディレクトリ
<作業ディレクトリの絶対パス>

## 担当タスク
<サブタスクの説明>

## コンテキスト
- プロジェクト概要: <調査で得た情報>
- 関連ファイル: <ファイルパスの一覧>
- 既存パターン: <コーディング規約・スタイル>

## ルール
- ファイルの読み書きは必ず作業ディレクトリの絶対パスを使う
- git 操作（add, commit）は行わない
- 他のエージェントの担当範囲のファイルは変更しない

## 先輩からのフィードバック
作業中に先輩エージェントから SendMessage でフィードバックが届くことがあります。
フィードバックが届いたら、内容を確認し、妥当であれば作業に反映してください。
```

#### 先輩エージェント（中間レビュー）

実装エージェントと**並列で**先輩エージェント（`subagent_type: general-purpose`）を **1 つ**起動する。
実装中のコードをリアルタイムで見て、問題があれば **SendMessage で各実装エージェントに直接フィードバック**する役割。

**重要**: 実装エージェントには必ず `name` パラメータを付ける（例: `name: "impl-api"`, `name: "impl-frontend"`）。
先輩エージェントにはそれらの name を伝え、SendMessage でフィードバックを送れるようにする。

先輩への指示：

```
## 役割
あなたは実装チームの先輩です。実装エージェントたちが作業中のコードをリアルタイムで見守り、問題があれば SendMessage で直接フィードバックしてください。

## 作業ディレクトリ
<作業ディレクトリの絶対パス>

## タスクの全体像
<元のタスク説明と実装計画の要約>

## 実装エージェント一覧（SendMessage の宛先）
- name: "impl-xxx" — <担当ファイル・タスク>
- name: "impl-yyy" — <担当ファイル・タスク>
- name: "impl-zzz" — <担当ファイル・タスク>

## レビュー手順
以下のループを、すべての実装エージェントが完了するまで繰り返す：

1. 60秒 sleep して、実装エージェントが作業を進める時間を確保する
2. 作業ディレクトリの変更ファイルを確認する（git diff, Read ツール等）
3. 以下の観点でチェックする：
   - 明らかなバグや論理エラー
   - 既存コードとのスタイル・パターンの不一致
   - 担当者間のファイル整合性（API の型が合っているか等）
4. 問題を発見したら、**該当する実装エージェントに SendMessage で即座にフィードバック**する
   - SendMessage の `to` に実装エージェントの name を指定する
   - 指摘内容は具体的に（ファイルパス・行番号・修正案を含める）
5. ステップ 1 に戻り、次のチェックを行う

## ループの終了条件
- すべての実装エージェントの作業が完了している（変更が収束し、新しい差分がなくなった）ことを確認したら終了する
- 最低でも 3 ラウンドはチェックすること
- 最大 10 ラウンドまで（無限ループ防止）

## ルール
- ファイルは読み取りのみ。変更は行わない
- git 操作は行わない
- 細かいスタイルの指摘は不要。実装に影響する問題のみ指摘する
- SendMessage でのフィードバックは簡潔に、修正に必要な情報だけ伝える
- フィードバックがない場合もループは続ける（実装の進捗を見守る）

## 出力形式
最終結果を以下の形式で返す：

### レビューサマリー
- ラウンド数: N 回
- 送信したフィードバック数: N 件

### 指摘事項（未解決のもの）
- [ファイルパス:行] 指摘内容

すべて解決済みまたは指摘なしの場合は「確認完了：指摘事項なし」とだけ返す。
```

#### 実装エージェント側の対応

実装エージェントへの指示に以下を追加する：

```
## 先輩からのフィードバック
作業中に先輩エージェントから SendMessage でフィードバックが届くことがあります。
フィードバックが届いたら、内容を確認し、妥当であれば作業に反映してください。
```

#### 並列実行の制約

- 同じファイルを複数の実装エージェントが編集しないようにする
- ファイルの担当が重なる場合はフェーズを分ける
- 各実装エージェントには担当ファイルを明示する
- 先輩は読み取り専用で、ファイル変更を行わない

### 5. フェーズ 3 — 統合・仕上げ

全エージェントの作業完了後：

1. **結果の確認** — 変更されたファイルを一覧し、整合性を確認
2. **先輩の指摘を反映** — 先輩エージェントの指摘があれば対応する
3. **仕上げ（refactor）** — `/refactor` を実行し、コードの品質を改善する
4. **テスト・ビルド確認** — テストやビルドがあれば実行し、失敗があれば修正する

### 6. PR 作成 & レビュー（LGTM ループ）

#### 8-1. 初回 PR 作成

`/pr draft review` の手順に従って実行する。

- Draft PR を作成し、先生エージェントによるレビューが PR コメントに投稿される

#### 8-2. レビュー結果の判定

先生エージェントのレビュー結果を確認する：

- **🔴（重要な問題）の指摘がない** → LGTM とみなし、ステップ 7 へ進む
- **🔴 の指摘がある** → ステップ 6-3 へ

#### 8-3. 指摘の修正

指摘事項を修正する：

1. 🔴 の指摘を優先的に修正する（🟡 も可能であれば対応する）
2. 修正後、`/commit` の手順でコミットする
3. `git push` で PR を更新する

#### 8-4. 再レビュー

再度、`/pr review` の手順（PR の review ステップのみ）に従い先生エージェントを起動する。

- 先生エージェントには前回の指摘事項と修正内容も伝える
- レビュー結果を受け取ったら **ステップ 6-2** に戻る

このループは LGTM が出るまで繰り返す。ただし **最大 3 回** までとし、3 回目でも 🔴 が残る場合はその旨をユーザーに報告してステップ 7 へ進む。

### 7. 完了報告

```
## チーム作業完了

### タスク
<元のタスク説明>

### 実施内容
- <サブタスク1>: 完了
- <サブタスク2>: 完了
- ...

### 変更ファイル
<git diff --stat の結果>

### テスト結果
<テスト実行結果の要約>

### レビュー結果
<レビュー指摘の要約、または「指摘事項なし」>

### PR
<PR の URL>
```

## エージェント編成の例

### 小規模タスク（1-2 ファイル）

エージェントは 1 つで十分。先輩・先生も不要。

### 中規模タスク（3-5 ファイル）

```
[Explore] 調査 ──→ [Agent A] フロントエンド ──→ 統合 ──→ /commit & /pr ──→ [先生] PR レビュー
                 ├→ [Agent B] バックエンド
                 └→ [先輩] 中間レビュー
```

### 大規模タスク（6+ ファイル）

```
[Explore x2] 調査 ──→ [Agent A] 機能実装    ──→ 統合 ──→ /commit & /pr ──→ [先生] PR レビュー
                    ├→ [Agent B] API/データ層
                    ├→ [Agent C] テスト作成
                    └→ [先輩] 中間レビュー
```

## 注意事項

- 実装エージェントの数は多くても 3 まで。増やしすぎるとコンフリクトが増える
- **先輩**は実装と並列で 1 つ。実装エージェントが完了するまでレビューをループし続け、SendMessage でリアルタイムにフィードバックする
- **先生**は PR 作成後に 1 つ。成果物全体を俯瞰レビューし PR コメントに投稿する
- 各エージェントに作業ディレクトリの**絶対パス**を必ず渡す
- エージェントが失敗した場合は、そのサブタスクだけリトライする
- `gh` CLI がインストール・認証済みであることが前提
