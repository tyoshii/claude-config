# /teamdev <タスク説明>

worktree を作成し、エージェントチームを編成して自律的に開発を進める。

ユーザーはタスクを投げたら待つだけ。
エージェントたちが調査・設計・実装・テストを分担して進める。

## 実行手順

### 1. worktree の判定

現在のディレクトリが worktree 内かどうかを判定する：

```bash
git rev-parse --is-inside-work-tree && test "$(git rev-parse --show-toplevel)" != "$(git rev-parse --path-format=absolute --git-common-dir | sed 's/\/\.git$//')"
```

- **worktree 内の場合**: ステップ 2 をスキップし、現在のディレクトリをそのまま作業ディレクトリとする
- **worktree 外（メインリポジトリ）の場合**: ステップ 2 で worktree を作成する

### 2. worktree の作成（メインリポジトリにいる場合のみ）

ブランチ名をタスク説明から自動生成する：

- 形式: `team/<短い英語の要約>`
- 例:
  - 「認証機能を追加して」→ `team/add-auth`
  - 「パフォーマンス改善」→ `team/improve-performance`

```bash
eval "$(command vibe start <branch>)" && pwd
```

- 出力の最後の行が新しい作業ディレクトリのパス
- 以降すべてのファイル操作はこのパスの絶対パスを使う
- Bash コマンドは `cd <worktree-path> && ...` で実行する

### 3. タスクの分解

タスク説明を分析し、並列に進められるサブタスクに分解する。

分解の方針：

- **依存関係がないもの**は並列に進める
- **依存関係があるもの**はフェーズを分けて順次実行
- 各サブタスクは 1 エージェントが完結できる粒度にする

### 4. .claude/rules の確認

作業に入る前に、worktree 内の `.claude/rules/` ディレクトリを確認する。

1. `.claude/rules/` 配下のファイルを Glob で一覧する
2. 各ルールファイルを読み、内容を把握する
3. 以降のすべてのフェーズで、ルールに従って作業を進める

ルールファイルが存在しない場合はスキップして次に進む。

### 5. フェーズ 1 — 調査

まず Agent ツール（`subagent_type: Explore`）を**並列に**使い、以下を把握する：

- 既存コードの構造と関連ファイル
- 使われている技術スタック・パターン
- テストの有無と実行方法
- CLAUDE.md やプロジェクト固有のルール

調査結果をもとに、実装計画を組み立てる。

### 6. フェーズ 2 — 実装

Agent ツール（`subagent_type: general-purpose`）を使い、サブタスクごとにエージェントを起動する。
**同時に先輩エージェントも起動する。**

#### 実装エージェントへの指示テンプレート

各実装エージェントには以下を伝える：

```
## 作業ディレクトリ
<worktree の絶対パス>

## 担当タスク
<サブタスクの説明>

## コンテキスト
- プロジェクト概要: <調査で得た情報>
- 関連ファイル: <ファイルパスの一覧>
- 既存パターン: <コーディング規約・スタイル>

## ルール
- ファイルの読み書きは必ず作業ディレクトリの絶対パスを使う
- git 操作（add, commit）は行わない
- 他のエージェントの担当範囲のファイルは変更しない
```

#### 先輩エージェント（中間レビュー）

実装エージェントと**並列で**先輩エージェント（`subagent_type: general-purpose`）を **1 つ**起動する。
実装中のコードをちょいちょい見て、問題があれば早めにフィードバックする役割。

先輩への指示：

```
## 役割
あなたは実装チームの先輩です。実装エージェントたちが作業中のコードを見守り、気になる点があれば早めにフィードバックしてください。

## 作業ディレクトリ
<worktree の絶対パス>

## タスクの全体像
<元のタスク説明と実装計画の要約>

## 担当者と担当範囲
- Agent A: <担当ファイル・タスク>
- Agent B: <担当ファイル・タスク>
- Agent C: <担当ファイル・タスク>

## レビュー手順
1. 30秒ほど待ってから、変更されたファイルを確認する
2. 以下の観点で軽くチェックする：
   - 明らかなバグや論理エラー
   - 既存コードとのスタイル・パターンの不一致
   - 担当者間のファイル整合性（API の型が合っているか等）
3. 問題を発見したらメモする
4. 間隔をおいてもう一度チェックし、新しい変更があれば確認する
5. 最終的なフィードバックをまとめて返す

## ルール
- ファイルは読み取りのみ。変更は行わない
- git 操作は行わない
- レビューは 3-5 回繰り返す（実装の進行に合わせて）
- 細かいスタイルの指摘は不要。実装に影響する問題のみ指摘する
- 最終レポートには指摘事項のみ記載する（「問題なし」の項目は不要）

## 出力形式
最終結果を以下の形式で返す：

### 指摘事項
- [ファイルパス:行] 指摘内容

指摘がない場合は「確認完了：指摘事項なし」とだけ返す。
```

#### 並列実行の制約

- 同じファイルを複数の実装エージェントが編集しないようにする
- ファイルの担当が重なる場合はフェーズを分ける
- 各実装エージェントには担当ファイルを明示する
- 先輩は読み取り専用で、ファイル変更を行わない

### 7. フェーズ 3 — 統合・仕上げ

全エージェントの作業完了後：

1. **結果の確認** — 変更されたファイルを一覧し、整合性を確認
2. **先輩の指摘を反映** — 先輩エージェントの指摘があれば対応する
3. **仕上げ（simplify）** — `/simplify` を実行し、コードの品質を改善する
4. **テスト・ビルド確認** — テストやビルドがあれば実行し、失敗があれば修正する

### 8. PR 作成

コミットして PR を作成する：

1. `git add` で変更ファイルをステージング（機密ファイルを除く）
2. `/commit` の手順に従ってコミットメッセージを作成・コミット
3. `git push -u origin <ブランチ名>` でプッシュ
4. `gh pr create --draft` で Draft PR を作成

PR 本文の形式：
```markdown
## Summary
- 変更点を箇条書きで簡潔に

## Test plan
- テスト方法や確認事項を箇条書きで
```

### 9. PR レビュー（先生エージェント）

PR 作成後、Agent ツール（`subagent_type: general-purpose`）で**先生エージェント**を起動する。
先輩とは別の視点で、成果物全体を俯瞰してレビューし、結果を PR コメントに投稿する。

先生への指示：

```
## 役割
あなたはシニアエンジニアとして、PR の最終レビューを行います。
実装の細部ではなく、設計・アーキテクチャ・全体の整合性を重視してレビューしてください。

## PR 情報
- PR URL: <作成した PR の URL>
- リポジトリ: <リポジトリのパス>
- ブランチ: <ブランチ名>

## 手順
1. `gh pr diff <PR番号>` で差分を取得
2. 変更内容を詳細にレビューする：
   - 設計・アーキテクチャ上の問題
   - バグや論理エラー
   - セキュリティ上の問題
   - エッジケースの考慮漏れ
   - テストの十分性
3. 指摘事項を PR にコメントとして投稿する

## コメント投稿方法

**全体コメント（総評）：**
gh pr comment <PR番号> --body "<レビュー総評>"

**ファイル単位のレビューコメント：**
gh api repos/<owner>/<repo>/pulls/<PR番号>/reviews \
  --method POST \
  -f body="コードレビュー" \
  -f event="COMMENT" \
  -f 'comments[][path]=<ファイルパス>' \
  -f 'comments[][position]=<diff内の行位置>' \
  -f 'comments[][body]=<コメント内容>'

※ `position` は diff のハンク内での行番号（1始まり）。
正確な position の算出が難しい場合は、全体コメントにファイル名と行番号を含めて投稿する。

## ルール
- 軽微なスタイルの指摘は省略してよい
- 重要な問題には 🔴、改善提案には 🟡、良い点には 🟢 のプレフィックスをつける
- git push や PR のマージは行わない
```

### 10. 完了報告

worktree に留まったまま報告する。ユーザーが確認・修正作業を続けられるようにする。

```
## チーム作業完了

### タスク
<元のタスク説明>

### 実施内容
- <サブタスク1>: 完了
- <サブタスク2>: 完了
- ...

### 変更ファイル
<git diff --stat の結果>

### テスト結果
<テスト実行結果の要約>

### レビュー結果
<レビュー指摘の要約、または「指摘事項なし」>

### PR
<PR の URL>

### 現在地
worktree: <worktree-path>（ブランチ: <ブランチ名>）

### 片付け
作業が終わったら以下で worktree を削除できます：
vibe clean
```

## エージェント編成の例

### 小規模タスク（1-2 ファイル）

エージェントは 1 つで十分。先輩・先生も不要。

### 中規模タスク（3-5 ファイル）

```
[Explore] 調査 ──→ [Agent A] フロントエンド ──→ 統合 ──→ /commit & /pr ──→ [先生] PR レビュー
                 ├→ [Agent B] バックエンド
                 └→ [先輩] 中間レビュー
```

### 大規模タスク（6+ ファイル）

```
[Explore x2] 調査 ──→ [Agent A] 機能実装    ──→ 統合 ──→ /commit & /pr ──→ [先生] PR レビュー
                    ├→ [Agent B] API/データ層
                    ├→ [Agent C] テスト作成
                    └→ [先輩] 中間レビュー
```

## 注意事項

- worktree 内で Claude を起動した場合（`vibe start` → `claude`）、cwd が worktree なので自然に動作する
- メインリポジトリから実行した場合は cwd が変わらないため、worktree 内の操作は絶対パスで行う
- 実装エージェントの数は多くても 3 まで。増やしすぎるとコンフリクトが増える
- **先輩**は実装と並列で 1 つ。実装中のコードを見守り早めにフィードバックする
- **先生**は PR 作成後に 1 つ。成果物全体を俯瞰レビューし PR コメントに投稿する
- 各エージェントに worktree の**絶対パス**を必ず渡す
- エージェントが失敗した場合は、そのサブタスクだけリトライする
- worktree 内での作業なので、メインの作業ディレクトリには影響しない
- `gh` CLI がインストール・認証済みであることが前提
